# AGENTS.md

## 适用范围

- 作用域：环境变量、构建、签名、打包、更新与交付验证规则
- 触发场景：涉及 `tauri.conf.json`、环境变量、打包、签名、自动更新、发布链路时阅读

### 阅读摘要
- 建议阅读：改构建命令、改环境变量、改签名、改打包配置、改自动更新链路
- 可先跳过：纯界面交互、纯业务逻辑、仅设计文档维护
- 优先查看：构建规则、交付规则、`提交前最小回归`

### 常见任务入口
- 改构建或打包配置：先看构建规则与交付验证要求
- 改环境变量、配置项、数据库路径：先看环境变量与运行约束
- 改签名、更新、发布链路：先看签名 / 更新规则与验证要求
- 补交付回归：先看 `提交前最小回归`

## 技术选择原则
如果仓库已经有真实实现，以现有代码为准，不要强行重构或替换技术栈。

**技术债务评估框架**：
- **沿用现有实现**：代码可正常运行、无明显性能问题、维护成本可接受、团队熟悉度高
- **必须重构**：存在安全漏洞、严重影响性能、阻碍新功能开发、维护成本过高
- **可选重构**：代码风格不一致、存在更好的替代方案、但不紧急
- **禁止重构**：仅为个人偏好、追赶技术潮流、非关键路径的过度优化

**重构决策流程**：
1. 先评估现有代码的技术债务等级（高 / 中 / 低）。
2. 判断重构的紧急程度和业务价值。
3. 评估重构风险和工作量。
4. 高风险或大规模重构需先与团队确认。

## 默认构建与发布约束

- 优先复用现有 `tauri.conf.json`、前端构建脚本与 Rust profile 配置
- 桌面应用默认不需要 Docker 作为主交付链路
- 发布链路必须可追踪、可复现
- 不要把密钥、签名材料、更新私钥写入仓库
- 若宿主环境与目标平台不一致，必须明确说明交叉编译或目标平台限制

---

## 环境变量规则

- 外部服务地址、Token、签名信息、更新地址等必须通过环境变量或安全配置注入
- `.env.example` 只保留非敏感示例值
- 区分：
  - 前端构建期变量
  - Tauri / Rust 运行期变量
  - 发布 / 签名阶段变量
- 新增环境变量时，必须同步文档与示例配置

---

## 签名、更新与打包规则

- 只要项目启用了自动更新，就必须把更新地址、签名流程和回滚策略写进文档
- 只要项目启用了签名，就必须明确区分：
  - 本地开发构建
  - CI 构建
  - 正式签名发布
- 更新元数据、安装包与版本号必须保持一致
- 不要为了“先能跑”而跳过签名 / 更新校验后直接把临时方案固化进模板

---

## 构建与交付规则

### 总体要求
- 前端构建与 Tauri 构建都必须可独立执行
- 优先使用仓库已有脚本组织构建流程，避免把复杂命令散在文档正文
- 发布前至少验证一次最终运行产物

### 交付验证至少包括
- 前端构建是否成功
- Rust / Tauri 编译是否成功
- 应用是否可启动
- 核心本地能力是否可用
- 若启用了数据库、更新或签名，对应链路是否已同步验证或明确说明未验证原因

### 提交前最小回归
- 修改环境变量、配置项或连接串时：确认 `.env.example`、文档和运行方式同步更新
- 修改 `tauri.conf.json`、构建脚本、打包脚本、签名或更新配置时：至少执行一次构建与启动验证
- 修改数据库目录、应用路径、缓存路径或文件权限策略时：至少补一项真实链路或等价验证
- 若本次改动同时影响应用访问链路，应至少补一次核心链路检查
