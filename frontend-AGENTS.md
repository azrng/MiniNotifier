# AGENTS.md

## 适用范围

- 作用域：前端实现、页面、组件、路由、状态、样式与前端测试
- 触发场景：涉及 React 页面、布局、组件、交互、样式、前端 smoke test 时阅读

### 阅读摘要
- 建议阅读：新增页面、改表单、改交互、改路由、改组件样式、补前端验证
- 可先跳过：纯 Rust / Tauri 逻辑、纯数据访问、仅文档整理、仅打包配置调整
- 优先查看：前端规则、状态 / 表单规则、测试规则

### 常见任务入口
- 新增页面或布局：先看 React / 路由规范与组件规则
- 改表单、筛选、交互：先看表单规则、状态管理与数据获取规则
- 改样式、主题、组件展示：先看样式规则与设计系统约束
- 补前端回归：先看 `提交前最小回归` 与测试规则

## 技术栈

### 前端
- React + TypeScript + Vite
- Tailwind CSS（唯一样式体系）
- Radix UI Primitive + `class-variance-authority` + `clsx` / `tailwind-merge`（推荐的组件基础设施）
- React Router（页面与导航）
- TanStack Query（异步数据状态）
- React Hook Form + Zod（表单与校验）
- Zustand（仅用于客户端跨组件共享状态）
- `@tauri-apps/api`（与 Tauri Runtime 通信）

### 测试
- Vitest + Testing Library
- MSW（仅在需要模拟外部请求时使用）

### 设计系统
- 所有视觉规范以 `design-system.yaml` 为唯一事实来源
- 优先复用项目已有组件、布局、样式变量与交互模式
- 禁止 inline style，禁止硬编码颜色、间距、圆角、阴影、断点

如果仓库已经有真实实现，以现有代码为准，不要强行重构或替换技术栈。

**技术债务评估框架**：
- **沿用现有实现**：代码可正常运行、无明显性能问题、维护成本可接受、团队熟悉度高
- **必须重构**：存在安全漏洞、严重影响性能、阻碍新功能开发、维护成本过高
- **可选重构**：代码风格不一致、存在更好的替代方案、但不紧急
- **禁止重构**：仅为个人偏好、追赶技术潮流、非关键路径的过度优化

**重构决策流程**：
1. 先评估现有代码的技术债务等级（高 / 中 / 低）。
2. 判断重构的紧急程度和业务价值。
3. 评估重构风险和工作量。
4. 高风险或大规模重构需先与团队确认。

## 推荐目录结构

若仓库尚未形成稳定结构，可优先参考以下组织方式；若仓库已有实现，以现状为准，不强制迁移。

```text
src/
├── app/                      # 路由与应用级入口
├── components/
│   ├── ui/                   # 基于 Radix / cva 的基础组件
│   ├── layout/               # 布局组件
│   └── business/             # 业务组件
├── features/                 # 按业务域收敛
├── hooks/                    # 自定义 Hook
├── routes/                   # 路由定义
├── lib/
│   ├── tauri/                # Tauri API 封装
│   ├── query/                # TanStack Query 封装
│   └── utils/                # 工具函数
├── stores/                   # Zustand store
├── schemas/                  # Zod schema
├── styles/                   # 全局样式与 token 映射
├── types/                    # TypeScript 类型
├── tests/                    # 前端测试辅助
├── main.tsx
└── App.tsx
```

## 阶段 1 — 前端实现

**触发条件**：用户发出“开始前端开发”或等价指令

**入场要求**：阶段 0 设计文档已确认

**工作内容**：
1. 基于 React + Vite 实现页面、布局、组件与导航结构。
2. 页面内允许先用 mock 数据、假 Command 响应或本地 fixture 演示交互。
3. 同步产出前端契约说明：字段定义、筛选条件、分页参数、表单校验规则、权限要求。
4. 只要仓库未明确采用别的组件体系，优先走“Tailwind + Radix / shadcn 风格组件”的收敛路线。

**门控规则**：
- 用户确认页面符合设计文档预期。
- 页面交互逻辑经手工 smoke test 或自动化验证通过。

## 前端规则

### React / 路由规范
- 函数组件 + Hooks 是唯一写法，禁止 class 组件
- 路由配置集中在 `routes/` 或 `app/`，目录命名必须表达业务语义
- 页面跳转统一通过 React Router 提供的 API 完成
- 页面级逻辑优先收敛在业务目录，不要把跨页面逻辑散落在全局
- 可复用逻辑放在 `hooks/`，优先按业务域收敛

### 数据获取与状态管理
- 异步数据默认优先使用 TanStack Query
- Query Key 使用稳定命名，禁止在组件内部临时拼字符串做缓存键
- 写操作优先封装成 mutation，再由 mutation 触发刷新或失效处理
- Zustand 仅用于主题、窗口状态、轻量全局筛选、全局通知、临时会话上下文等纯客户端状态
- 禁止把“可由 Query 管理的服务端 / 本地命令数据”长期放进 Zustand

### Tauri 前端调用规范
- 所有 Tauri 调用统一收敛在 `lib/tauri/` 或按业务域封装，不直接散落在页面组件中
- 优先使用 `invoke`、event、plugin API 的封装层，不在页面里直接拼 command 名称
- Command 参数与返回值应先经过 TypeScript 类型与 Zod schema 约束
- 除非明确需要，否则不要在 Tauri 桌面应用里再额外引入本地 HTTP 服务作为前后端桥接层
- 若确实需要流式处理、WebSocket、嵌入式代理等能力，再单独论证是否引入额外服务层

### 表单规则
- 表单优先使用 React Hook Form + Zod
- 表单 schema、默认值、字段映射尽量与 `schemas/` 中的 Zod 定义保持一致
- 错误提示要区分字段校验错误、权限错误、状态冲突、系统异常
- 禁止在组件中手写大量分散校验逻辑替代 schema

### 样式规则
- 统一使用 Tailwind CSS 与设计 token
- 基础组件优先通过 cva / 变体模式管理，不在业务组件里堆大量条件 class
- 禁止 inline style
- 禁止硬编码颜色值、阴影值、圆角值、z-index、断点
- 响应式遵循桌面优先，但仍需兼容窗口缩放和不同分辨率

### 组件规则
- 优先复用 `components/ui` 与业务域组件
- 只有确认存在复用价值时，才新增共享组件
- 页面状态必须完整：`loading`、`empty`、`error`、`no-permission`
- 不要为单次需求过度抽象组件

### 外部请求规则
- Tauri 本地能力优先通过 Command / plugin 调用，不通过本地 HTTP 绕路
- 如果需要访问外部服务，可使用 `fetch` 或统一封装的请求层
- 外部请求必须统一处理超时、错误和取消逻辑
- 若仓库已经使用 MSW，测试时优先通过 MSW 模拟外部请求

## 测试规则

### 提交前最小回归
- 默认执行：`lint`、`typecheck`
- 页面、路由、表单、交互改动：至少补一次受影响页面或组件的 smoke test / 交互验证
- 仅样式或文案微调：可不补自动化测试，但仍应检查受影响页面显示、状态和关键交互未回退
- 若改动影响共享组件、布局或状态流转，优先补受影响范围最大的页面验证，而不是只看局部组件

### 前端测试
- 组件测试优先使用 Vitest + Testing Library
- Query、表单、交互、禁用态、错误态应有验证
- 仅在需要模拟外部请求时才引入 MSW
- 对 Tauri 调用统一使用 mock 封装层，不直接 mock 页面内部实现细节

### 无法执行测试时
- 必须说明未执行的测试类型
- 必须说明未执行原因
- 必须说明潜在影响范围与风险
