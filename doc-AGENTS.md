# AGENTS.md

## 适用范围

- 作用域：设计文档、契约说明、设计系统与开发文档维护
- 触发场景：涉及阶段 0 设计、契约说明、`design-system.yaml`、`doc/requirement.md`、`doc/devlog/`、`doc/task-archive/` 时阅读

### 阅读摘要
- 建议阅读：整理需求、补设计文档、维护契约、更新设计系统、记录 `doc/devlog/`
- 可先跳过：纯界面实现、纯 Rust / Tauri 逻辑、纯打包配置调整
- 优先查看：阶段 0 设计文档、契约说明规范、文档更新要求

### 常见任务入口
- 新增功能方案：先看阶段 0 设计文档
- 更新 Command、DTO、schema 或事件载荷：先看契约说明规范
- 更新设计系统或通用说明：先看设计系统文件规范与文档目录约定
- 补开发记录、任务归档或回顾：先看 devlog、task archive 与回顾分析记录格式

---

## 阶段 0 设计文档

### 阶段 0 — 设计文档（新功能强制前置）

**触发条件**：用户提出新功能需求

**执行方式**：由用户与 AI 协作产出，用户最终确认定稿

**产物**：`doc/design/设计文档.md`（全项目一份，新功能以新增章节方式补充）

**文档必须包含以下四个部分**：

| 部分 | 内容要求 |
| --- | --- |
| 架构设计 | 模块划分、Tauri / Rust / 前端边界、数据流向、缓存与权限方案 |
| 功能需求 | 页面清单、功能点明细、业务规则、权限矩阵 |
| 界面原型 | 每个页面的 ASCII 线框图，包含 `loading` / `empty` / `error` / `no-permission` 状态 |
| 交互说明 | 操作流程、状态流转、边界场景、错误提示与恢复方式 |

**门控规则**：
- 用户明确确认设计文档后，才允许进入阶段 1
- AI 在此阶段只输出文档内容，不编写实现代码

## 契约说明规范

### 契约要求

- **前端唯一事实来源**：`schemas/` 中的 Zod schema 与共享类型定义
- **Rust 边界事实来源**：`src-tauri/` 中的 DTO / Command 输入输出定义
- **定义方式**：表单 schema、Query schema、Command 入参 schema、Command 出参结构必须成对维护
- **变更规则**：字段或行为变化时，必须同步更新 schema、类型、页面、Command 封装与测试
- **调用选择原则**：
  - 桌面应用内部业务交互：优先 Tauri Command
  - 事件通知或后台状态推送：使用 event
  - 外部服务访问：使用前端统一请求层或 Rust 侧 integration，不新增本地 HTTP 桥接

## 设计系统文件规范

`design-system.yaml` 是唯一设计系统文件，包含：
- 品牌与产品风格意图
- 颜色、排版、间距、圆角、阴影等 token
- 布局与断点规则
- 页面状态规范
- 表单与反馈组件规则
- 明确的 `do / dont`

## 文档更新要求

以下内容发生变化时，必须同步更新相关文档：
- 命令、脚本或启动方式
- 环境变量
- 配置项
- 构建、发布、签名或更新流程
- 本地数据库结构、迁移或初始化方式
- 对外使用方式或接口约定

优先更新已有文档，不新增重复说明。
每次开发完成后，必须在 `doc/devlog/` 下补充一份简短开发记录。

### 文档目录约定
- `doc/requirement.md`：需求文档。若文件已存在，AI 应基于用户新增需求进行补充或修订；若文件不存在，AI 必须根据用户输入先自动生成详细初稿。该文件仅承载业务需求，不写入设计说明或开发过程记录
- `doc/design/设计文档.md`：项目设计文档，全项目一份。包含架构设计、功能需求、界面原型与交互说明。由用户与 AI 协作产出，用户确认后作为开发的唯一需求基准。AI 可在文件中追加新功能章节，不删除已有内容，不新建平行文件
- `doc/devlog/`：每次开发完成后的简短开发日志
- `doc/task-archive/`：`TASK.md` 中较早的 `DONE` 任务月度归档目录，不记录仍需处理的活动任务

### requirement 要求
- 若 `doc/requirement.md` 为模板初始状态，AI 应根据用户输入将占位内容替换为项目真实需求，而不是长期保留“待补充”空壳。
- 需求文档建议固定包含以下章节：项目背景、目标与范围、用户角色、页面或模块清单、核心业务流程、关键数据与字段、权限与可见范围、边界场景与异常、非功能要求、待确认项。
- 自动生成需求文档时，优先写清业务事实、流程与边界，不提前写入技术实现方案。

### devlog 要求
- 文件目录：`doc/devlog/`
- 文件格式：Markdown
- 文件命名建议：`YYYY-MM-DD-HH-mm-ss-中文任务简述.md`
- 文件名中的任务简述必须使用中文，不使用英文 slug、拼音缩写或中英混杂描述
- 内容保持简短，至少包含：
  - 本次目标
  - 核心改动
  - 修改文件
  - 校验情况
  - 风险或遗留项

### task archive 要求
- 文件目录：`doc/task-archive/`
- 文件命名建议：`YYYY-MM.md`
- 仅归档 `DONE` 任务，不归档 `TODO`、`DOING`、`BLOCKED`、`REVIEW`
- 归档文件建议保留原任务表字段，并可增加 `归档日期`
- `TASK.md` 中应保留归档索引，指向对应月度归档文件

### 回顾分析记录格式
- 适用场景：同类错误连续出现两次后，触发简短回顾分析时使用
- 内容保持简短，建议至少包含：
  - 错误现象
  - 触发场景
  - 根因
  - 本次补充或修改的规则
  - 后续如何避免或验证不再复发
