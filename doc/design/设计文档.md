## 1. 喝水提醒托盘应用

### 1.1 架构设计

#### 1.1.1 模块划分

- `App` 启动层：负责单实例启动、`Generic Host`、依赖注入、托盘组件初始化、窗口生命周期管理。
- `Views`
  - `HydrationSettingsWindow`：唯一主操作窗口，用于查看和修改提醒设置，采用现代化 Fluent 卡片布局。
  - `HydrationReminderWindow`：右下角提醒弹窗，支持手动关闭和自动关闭，采用轻玻璃感浮层样式。
- `ViewModels`
  - `HydrationSettingsViewModel`：维护设置页状态、保存设置、立即测试提醒。
  - `HydrationReminderViewModel`：维护提醒文案、倒计时显示、关闭动作。
- `Services`
  - `ITrayService`：托盘图标、菜单、双击打开设置、退出应用。
  - `IReminderSchedulerService`：基于 `DispatcherTimer` 进行提醒调度。
  - `IReminderPopupService`：统一创建和关闭提醒弹窗。
  - `ISettingsService`：读写提醒配置，负责默认值回填。
  - `IReminderMessageService`：根据时间段和内置文案池随机生成提醒标题与正文。
  - `IMouseActivityService`：统计近期全局鼠标点击活跃度，用于辅助推断当前工作状态。
- `Repositories`
  - `ISettingsRepository`：基于 JSON 文件持久化提醒设置。
- `Models`
- `HydrationSettings`：提醒间隔、弹窗自动关闭秒数、启用状态、最近提醒时间等领域模型。
- `HydrationSettingsDto`：阶段 1 输出的稳定契约。
- `HydrationSettingsJsonContext`：`System.Text.Json` source generation 上下文，保障发布兼容性。
  - `StartupSettingsDto`：开机自启动设置契约，供阶段 2 接入真实系统能力。

#### 1.1.2 数据流向

1. 应用启动后创建托盘图标，不主动显示主窗口。
2. `ISettingsService` 读取本地 JSON 配置；若文件不存在，则回填默认配置：
   - 喝水提醒开启
   - 提醒间隔 30 分钟
   - 弹窗自动关闭 15 秒
3. `IReminderSchedulerService` 使用 `DispatcherTimer` 按“下一次提醒时间”动态调度，而不是固定 1 秒轮询。
4. 到达提醒时间后，由 `IReminderPopupService` 打开右下角提醒弹窗。
5. 用户可通过弹窗关闭按钮关闭，也可等待 15 秒自动关闭。
6. 用户在设置窗口修改配置并保存后，`ISettingsService` 写入 JSON，调度服务即时重载。
7. 用户通过托盘菜单选择“退出应用”时，先释放计时器和托盘资源，再退出进程。

#### 1.1.3 技术决策

- 仅面向 Windows，采用 `.NET 10 + WPF + Wpf.Ui + CommunityToolkit.Mvvm`。
- 使用 `Microsoft.Extensions.Hosting` 的 `Generic Host` 管理 DI、配置和应用生命周期。
- 托盘采用 `Hardcodet.NotifyIcon.Wpf`，以获得更贴近 WPF 的托盘集成能力。
- 定时调度使用 `DispatcherTimer`，但按“距离下次提醒剩余时间”动态设置 Tick 间隔，减少空轮询。
- 提醒弹窗使用独立 WPF Window，自带显示动画和自动关闭计时器，优先保证样式可控和视觉一致性。
- 配置持久化采用 `System.Text.Json + source generation`，文件存放于 `%LocalAppData%/MiniNotifier/hydration-settings.json`。
- 开机自启动采用当前用户注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`，避免要求管理员权限。
- 提醒文案采用内置多套模板池，按晨间 / 午间 / 下午 / 傍晚 / 深夜等时间段随机生成，风格允许轻松搞笑，但不使用悲伤语气。
- 在不接入天气和节假日能力的前提下，可基于近期鼠标点击频率推断“深度专注 / 平稳推进 / 高频处理”等工作状态，用于调整提醒语气。
- 发布采用“框架依赖 + 单文件”模式，优先减少发布包体积；本阶段不启用 Trim，避免与 framework-dependent 模式冲突。
- 为避免重复提醒和资源浪费，应用需支持单实例运行。

#### 1.1.4 视觉与交互风格

- 整体视觉采用 `Fluent + Glass + Soft Glow` 方向，而不是传统系统设置页。
- 设置窗口使用顶部品牌区、中央主卡片、底部辅助说明的三段式布局，强调信息层次。
- 弹窗采用圆角浮层、轻微透明背景、柔和描边和右下角滑入动效，营造“轻提醒”体验。
- 重点信息用大号数字和状态标签呈现，例如“30 分钟”“15 秒自动关闭”“下一次 19:30”。
- 动效保持克制：
  - 窗口进入使用 `180-220ms` 位移 + 透明度过渡
  - 表单状态切换使用淡入淡出
  - 保存成功使用轻量 `Snackbar`
- 现代化不等于复杂化，所有主要操作必须在单窗口内完成，不新增无意义层级。

### 1.2 功能需求

#### 1.2.1 页面与入口

| 页面 / 入口 | 说明 |
| ----------- | ---- |
| 系统托盘图标 | 应用主入口，启动后仅显示托盘图标，不在任务栏保留主窗口 |
| 设置窗口 | 查看与修改喝水提醒配置，展示当前状态、下次提醒时间和现代化状态卡片 |
| 提醒弹窗 | 右下角浮出提醒窗口，展示喝水提示、倒计时和关闭操作 |

#### 1.2.2 功能点明细

1. 后台与托盘
   - 应用启动后默认最小化到托盘，不主动弹出主窗口。
   - 托盘图标悬停时显示应用名称与当前提醒状态。
   - 托盘菜单支持打开设置、立即提醒一次、退出应用。
   - 双击托盘图标打开设置窗口。

2. 喝水提醒配置
   - 默认提醒间隔为 30 分钟。
   - 用户可修改提醒间隔，单位为分钟。
   - 用户可开启或关闭喝水提醒。
   - 用户可开启或关闭开机自启动。
   - 用户可修改弹窗自动关闭时长，默认 15 秒。
   - 保存后立即生效，并持久化到 JSON 文件。

3. 提醒弹窗
   - 到达提醒时间时在屏幕右下角显示。
   - 弹窗显示随机提醒标题、随机正文、关闭按钮。
   - 文案根据当前时间段动态切换，可使用搞笑、逗趣、轻松的表达方式，但不允许出现悲伤语气。
   - 文案可参考近期鼠标点击活跃度，适当判断当前更像是深度专注还是高频处理状态。
   - 用户未操作时，倒计时结束自动关闭。
   - 用户点击关闭后立即消失，不影响下一轮提醒。

4. 状态展示
   - 设置窗口展示：
     - 当前提醒是否启用
     - 当前提醒间隔
     - 当前自动关闭秒数
     - 当前是否开机自启动
     - 上次提醒时间
     - 下次预计提醒时间
     - 最近保存状态
   - 当配置文件不存在时，以默认值初始化并提示“已加载默认配置”。

5. 现代化 UI 表现
   - 主界面左上角必须展示应用图标、应用名称和一句简短描述。
   - 首页使用至少一块核心状态卡片，强调“提醒中 / 已暂停 / 下次提醒时间”。
   - 重要操作按钮采用主次分明的视觉层级，不出现传统表单堆砌感。
   - 提醒弹窗加入柔和动效、玻璃感背景和大标题文案，整体观感偏现代而非传统 MessageBox。

#### 1.2.3 业务规则

- 提醒间隔允许范围：`5 - 240` 分钟。
- 自动关闭秒数允许范围：`3 - 15` 秒。
- 当提醒被关闭时，不再弹窗，但仍保留托盘图标和设置入口。
- 开机自启动默认为关闭，用户可独立控制，不受提醒总开关影响。
- 保存配置成功后，下一次提醒时间按“保存时刻 + 新间隔”重新计算。
- 立即提醒一次不会改写下一次正常提醒时间。
- 退出应用前必须释放 `NotifyIcon` 和所有定时器。
- 配置写入失败时，不更新内存中的已保存状态，并提示用户重试。

#### 1.2.4 关键状态

- `loading`：启动时读取配置，设置窗口进入加载态。
- `empty`：首次启动无配置文件，显示默认配置说明态。
- `error`：配置读取或保存失败，显示错误说明和重试按钮。
- `no-permission`：配置目录无法写入时，提示无权限并禁用保存按钮。

### 1.4 交互说明

#### 1.4.1 启动流程

1. 用户启动应用。
2. 应用检测单实例；若已有实例运行，则激活已有设置窗口或仅提示已在托盘运行。
3. 应用加载配置并创建托盘图标。
4. 若启动参数不要求打开界面，则主窗口不显示，直接进入后台运行。
5. 调度服务开始计时，等待下一次提醒。

#### 1.4.2 托盘交互流程

1. 用户右键托盘图标，打开菜单。
2. 选择“打开设置”后显示设置窗口。
3. 选择“立即提醒一次”后立即弹出提醒窗口。
4. 选择“暂停提醒 / 恢复提醒”后即时更新调度状态和托盘提示文本。
5. 选择“退出应用”后执行资源释放并结束进程。

#### 1.4.3 设置保存流程

1. 用户在设置窗口修改提醒开关、间隔或自动关闭时长。
2. 前端先执行表单校验：
   - 间隔必须为 5 至 240 的整数
   - 自动关闭必须为 3 至 15 的整数
   - 开机自启动为布尔开关，无额外数值校验
3. 校验通过后调用保存命令。
4. 保存成功：
   - 写入 JSON
   - 更新界面状态
   - 调度器重算下一次提醒时间
   - 弹出成功提示
5. 保存失败：
   - 保留用户输入
   - 显示错误提示
   - 不覆盖当前已生效配置

#### 1.4.4 提醒弹窗流程

1. 调度器检测到到期时间，触发提醒。
2. 弹窗从右下角进入，默认置顶但不抢占焦点。
3. 弹窗标题和正文根据当前时间段从内置文案池中随机生成。
4. 用户可以点击关闭按钮立即关闭。
5. 若无操作，则在倒计时结束后自动关闭。
6. 弹窗关闭后，调度器按提醒间隔计算下一次触发时间。
7. 若用户从托盘选择“立即提醒一次”，弹窗样式与常规提醒一致，但不重置正常调度时间。

#### 1.4.5 边界场景

- 首次运行无配置文件：自动生成默认配置并进入 empty 提示态。
- 配置文件损坏：回退到 error 态，允许用户重试或恢复默认值。
- 系统时间被修改：下一轮检查按“当前时间 >= 下次提醒时间”重新判断。
- 设置窗口被关闭：应用继续后台运行，不退出。
- 用户禁用提醒：清除待触发提醒，仅保留托盘入口。
- 用户临时暂停提醒：保留已保存配置，但暂停当前调度。
- 无权限写入配置目录：进入 no-permission 态，允许只读查看当前内存配置。

#### 1.4.6 错误处理方式

- 配置读取失败：提示“无法读取提醒配置”，提供重试。
- 配置保存失败：提示“保存失败，当前设置未生效”，保留编辑内容。
- 托盘初始化失败：记录日志并中止启动，避免进入不可退出状态。
- 弹窗创建失败：记录日志，保留下一轮提醒，不让调度器崩溃。
- 托盘库异常释放失败：写日志后继续退出流程，避免进程残留。

## 2. MiniNotifier 从 WPF 渐进迁移到 Tauri

### 2.1 架构设计

#### 2.1.1 迁移目标

- 保留当前产品定位不变：`Windows 托盘喝水提醒工具`
- 迁移后采用 `Tauri 2 + Rust + React + TypeScript`
- 迁移原则不是“一次性重写所有内容”，而是：
  1. 先验证关键原生能力
  2. 再搭建前端设置页壳层
  3. 再替换持久化与调度等业务能力
  4. 最后完成联调、打包和切换交付

#### 2.1.2 当前状态与目标状态

| 维度 | 当前状态 | 目标状态 |
| ---- | -------- | -------- |
| 桌面壳 | WPF 主窗口 + WPF 提醒弹窗 | Tauri 窗口体系 |
| 原生能力 | C# / WPF + Windows API | Rust + Tauri plugin / Windows API |
| 设置界面 | WPF + Wpf.Ui | React + TypeScript |
| 持久化 | JSON 文件 | 优先 `Store / JSON`，按需升级 SQLite |
| 调度 | `DispatcherTimer` | Rust 后台调度 |
| 托盘 | `Hardcodet.NotifyIcon.Wpf` | Tauri tray / Rust 封装 |
| 开机自启 | Windows 注册表 | Tauri autostart plugin 或 Rust 注册表封装 |

#### 2.1.3 模块划分

- `src/` 前端层
  - `app/`：应用壳层与入口
  - `features/settings/`：设置页
  - `components/`：基础组件与业务组件
  - `lib/tauri/`：对 `invoke` / `event` 的统一封装
  - `schemas/`：表单和 DTO 对应的 schema
- `src-tauri/src/` 原生层
  - `commands/`：前端调用入口
  - `services/`：调度、提醒、设置保存、状态编排
  - `integrations/`：托盘、自启动、窗口唤起、全局鼠标监听、通知
  - `stores/`：配置持久化，初期默认 JSON / Store
  - `models/`：配置 DTO、提醒状态、弹窗载荷
  - `state/`：应用共享状态

#### 2.1.4 数据流向

1. 应用启动后，由 Rust 侧创建托盘、加载配置、初始化后台能力。
2. 启动时优先恢复本地设置，并计算下一次提醒时间。
3. 前端设置页打开后，通过 `Command` 读取当前配置和运行状态。
4. 用户修改设置后，前端调用保存命令，由 Rust 统一完成校验、持久化和调度刷新。
5. 到达提醒时间时，Rust 负责触发提醒弹窗窗口，并在关闭后重算下一次提醒时间。
6. 托盘菜单与设置页对同一份运行状态生效，避免前后端各自维护一份真相。

#### 2.1.5 技术决策

- 当前业务仍是单机桌面工具，不默认引入本地 HTTP 桥接层。
- 前后端通信统一使用 `Tauri Command + event`。
- 配置持久化第一阶段优先 `tauri-plugin-store` 或 JSON 文件，不默认上 SQLite。
- 只有在引入提醒历史、统计分析、多实体关系或复杂筛选时，才升级 SQLite。
- 前端默认采用最小技术栈：
  - `React + TypeScript + Vite`
  - `React Hook Form + Zod`
  - `Tailwind CSS`
  - 路由、Query、全局 Store 按需引入
- 对本项目最关键的能力不是页面，而是：
  - 托盘
  - 开机自启
  - 提醒弹窗
  - 后台调度
  - 全局鼠标监听
- 因此迁移前必须先做一次原生能力 spike，确认上述能力在 Tauri / Rust 下可稳定落地。

### 2.2 功能需求

#### 2.2.1 迁移范围

本次迁移目标包括：

1. 保留当前喝水提醒产品行为不变：
   - 托盘入口
   - 设置页
   - 提醒弹窗
   - 开机自启
   - 提醒配置持久化
   - 随机提醒文案
   - 鼠标活跃度辅助判断
2. 将可见设置界面迁移为 React。
3. 将系统集成与后台能力迁移为 Rust。
4. 保持重新启动应用后配置和行为仍真实生效。

#### 2.2.2 分阶段迁移功能点

1. 阶段 A：原生能力 spike
   - 验证托盘是否可稳定驻留
   - 验证窗口可被托盘重新唤起
   - 验证右下角提醒弹窗窗口可独立展示与关闭
   - 验证开机自启接入
   - 验证后台定时调度
   - 验证全局鼠标点击监听

2. 阶段 B：前端设置页壳层
   - 搭建 Tauri 前端工程
   - 实现单窗口设置页
   - 先用 mock 或假命令演示交互
   - 输出稳定 schema 与字段说明

3. 阶段 C：业务逻辑迁移
   - 接入真实配置读写
   - 接入真实保存、暂停、立即提醒
   - 接入下一次提醒时间计算
   - 接入随机提醒文案与工作状态判断

4. 阶段 D：联调与切换
   - 前后端绑定真实数据流
   - 完成发布构建
   - 完成关键路径 smoke test
   - 评估是否停用旧 WPF 入口

#### 2.2.3 业务规则

- 迁移后产品行为应尽量与现有 WPF 版保持一致，不借迁移之名擅改用户可感知规则。
- 在迁移前期允许“前端壳层先完成、原生能力分步接入”，但所有已接入的功能必须真实生效。
- 若某项原生能力在 Tauri 下暂时无法达到当前 WPF 体验，应记录差异，不可悄悄降级后直接替换。
- 配置结构变更时，需考虑兼容现有本地 JSON，避免老用户升级后丢失设置。

#### 2.2.4 关键状态

- `spike-pending`：关键原生能力尚未验证，不进入大规模页面开发
- `shell-ready`：前端设置页壳层已可运行，但仍为 mock 或半真半假数据
- `native-ready`：托盘、自启动、弹窗、调度等原生能力已完成
- `integrated`：前后端数据流打通，关键链路可真实执行
- `release-ready`：完成构建、启动与 smoke test，具备交付条件

### 2.4 交互说明

#### 2.4.1 迁移实施流程

1. 先创建 Tauri 迁移分支与迁移设计文档章节。
2. 先完成原生能力 spike，不急着重做全部 UI。
3. spike 通过后，再开始 React 设置页壳层开发。
4. 页面 schema 和命令契约稳定后，再接入 Rust 真实能力。
5. 联调完成后，进行构建、启动和 smoke test。
6. 最后评估是否切换默认入口、保留旧 WPF 版本回滚窗口。

#### 2.4.2 原生能力 spike 流程

1. 启动 Tauri 应用最小壳。
2. 验证托盘创建、菜单点击、窗口显示与退出流程。
3. 验证提醒弹窗窗口的展示位置、关闭方式、置顶与不抢焦点行为。
4. 验证开机自启开启/关闭能力。
5. 验证后台定时任务在窗口关闭后仍可继续工作。
6. 验证全局鼠标监听在目标环境中可用且可被关闭。
7. 若任一关键能力不达标，则先调整技术方案，不进入大规模前端开发。

#### 2.4.3 契约冻结流程

1. React 设置页先基于 mock 展示完整状态。
2. 与 Rust 侧确认命令入参、出参、事件载荷。
3. 若关键原生能力已稳定，则冻结第一版字段契约。
4. 再逐步把 mock 替换为真实命令调用。

#### 2.4.4 数据迁移与兼容

1. 启动迁移版时优先检查现有 WPF 版配置文件。
2. 若结构兼容，则直接读取并映射为新配置结构。
3. 若字段扩展，则补默认值，不清空旧用户配置。
4. 若配置损坏，则给出错误提示并允许恢复默认设置。

#### 2.4.5 风险与边界场景

- 原生窗口行为可能与 WPF 表现略有差异，例如弹窗焦点、阴影、动画和边框处理。
- 托盘和全局监听依赖 Windows 行为，需在真实目标环境验证，而不是只看开发机。
- 若鼠标监听能力在 Tauri / Rust 下接入成本过高，可先将其列为第二阶段增强项，但需明确记录。
- 若前端过早引入复杂路由和缓存层，容易掩盖真正的迁移风险，因此应延后。

#### 2.4.6 错误处理方式

- spike 失败：记录具体失败能力、现象、替代方案，不继续推进页面层。
- 配置兼容失败：进入错误态，提示恢复默认值或查看日志。
- 系统能力调用失败：Rust 侧记录详细日志，前端仅展示稳定错误文案。
- 构建成功但运行异常：归类为阶段 D 阻塞，不直接宣告迁移完成。
