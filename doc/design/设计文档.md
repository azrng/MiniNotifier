## 1. 喝水提醒托盘应用

### 1.1 架构设计

#### 1.1.1 模块划分

- `App` 启动层：负责单实例启动、`Generic Host`、依赖注入、托盘组件初始化、窗口生命周期管理。
- `Views`
  - `HydrationSettingsWindow`：唯一主操作窗口，用于查看和修改提醒设置，采用现代化 Fluent 卡片布局。
  - `HydrationReminderWindow`：右下角提醒弹窗，支持手动关闭和自动关闭，采用轻玻璃感浮层样式。
- `ViewModels`
  - `HydrationSettingsViewModel`：维护设置页状态、保存设置、立即测试提醒。
  - `HydrationReminderViewModel`：维护提醒文案、倒计时显示、关闭动作。
- `Services`
  - `ITrayService`：托盘图标、菜单、双击打开设置、退出应用。
  - `IReminderSchedulerService`：基于 `DispatcherTimer` 进行提醒调度。
  - `IReminderPopupService`：统一创建和关闭提醒弹窗。
  - `ISettingsService`：读写提醒配置，负责默认值回填。
  - `IReminderMessageService`：根据时间段和内置文案池随机生成提醒标题与正文。
  - `IMouseActivityService`：统计近期全局鼠标点击活跃度，用于辅助推断当前工作状态。
- `Repositories`
  - `ISettingsRepository`：基于 JSON 文件持久化提醒设置。
- `Models`
- `HydrationSettings`：提醒间隔、弹窗自动关闭秒数、启用状态、最近提醒时间等领域模型。
- `HydrationSettingsDto`：阶段 1 输出的稳定契约。
- `HydrationSettingsJsonContext`：`System.Text.Json` source generation 上下文，保障发布兼容性。
  - `StartupSettingsDto`：开机自启动设置契约，供阶段 2 接入真实系统能力。

#### 1.1.2 数据流向

1. 应用启动后创建托盘图标，不主动显示主窗口。
2. `ISettingsService` 读取本地 JSON 配置；若文件不存在，则回填默认配置：
   - 喝水提醒开启
   - 提醒间隔 30 分钟
   - 弹窗自动关闭 15 秒
3. `IReminderSchedulerService` 使用 `DispatcherTimer` 按“下一次提醒时间”动态调度，而不是固定 1 秒轮询。
4. 到达提醒时间后，由 `IReminderPopupService` 打开右下角提醒弹窗。
5. 用户可通过弹窗关闭按钮关闭，也可等待 15 秒自动关闭。
6. 用户在设置窗口修改配置并保存后，`ISettingsService` 写入 JSON，调度服务即时重载。
7. 用户通过托盘菜单选择“退出应用”时，先释放计时器和托盘资源，再退出进程。

#### 1.1.3 技术决策

- 仅面向 Windows，采用 `.NET 10 + WPF + Wpf.Ui + CommunityToolkit.Mvvm`。
- 使用 `Microsoft.Extensions.Hosting` 的 `Generic Host` 管理 DI、配置和应用生命周期。
- 托盘采用 `Hardcodet.NotifyIcon.Wpf`，以获得更贴近 WPF 的托盘集成能力。
- 定时调度使用 `DispatcherTimer`，但按“距离下次提醒剩余时间”动态设置 Tick 间隔，减少空轮询。
- 提醒弹窗使用独立 WPF Window，自带显示动画和自动关闭计时器，优先保证样式可控和视觉一致性。
- 配置持久化采用 `System.Text.Json + source generation`，文件存放于 `%LocalAppData%/MiniNotifier/hydration-settings.json`。
- 开机自启动采用当前用户注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`，避免要求管理员权限。
- 提醒文案采用内置多套模板池，按晨间 / 午间 / 下午 / 傍晚 / 深夜等时间段随机生成，风格允许轻松搞笑，但不使用悲伤语气。
- 在不接入天气和节假日能力的前提下，可基于近期鼠标点击频率推断“深度专注 / 平稳推进 / 高频处理”等工作状态，用于调整提醒语气。
- 发布采用“框架依赖 + 单文件”模式，优先减少发布包体积；本阶段不启用 Trim，避免与 framework-dependent 模式冲突。
- 为避免重复提醒和资源浪费，应用需支持单实例运行。

#### 1.1.4 视觉与交互风格

- 整体视觉采用 `Fluent + Glass + Soft Glow` 方向，而不是传统系统设置页。
- 设置窗口使用顶部品牌区、中央主卡片、底部辅助说明的三段式布局，强调信息层次。
- 弹窗采用圆角浮层、轻微透明背景、柔和描边和右下角滑入动效，营造“轻提醒”体验。
- 重点信息用大号数字和状态标签呈现，例如“30 分钟”“15 秒自动关闭”“下一次 19:30”。
- 动效保持克制：
  - 窗口进入使用 `180-220ms` 位移 + 透明度过渡
  - 表单状态切换使用淡入淡出
  - 保存成功使用轻量 `Snackbar`
- 现代化不等于复杂化，所有主要操作必须在单窗口内完成，不新增无意义层级。

### 1.2 功能需求

#### 1.2.1 页面与入口

| 页面 / 入口 | 说明 |
| ----------- | ---- |
| 系统托盘图标 | 应用主入口，启动后仅显示托盘图标，不在任务栏保留主窗口 |
| 设置窗口 | 查看与修改喝水提醒配置，展示当前状态、下次提醒时间和现代化状态卡片 |
| 提醒弹窗 | 右下角浮出提醒窗口，展示喝水提示、倒计时和关闭操作 |

#### 1.2.2 功能点明细

1. 后台与托盘
   - 应用启动后默认最小化到托盘，不主动弹出主窗口。
   - 托盘图标悬停时显示应用名称与当前提醒状态。
   - 托盘菜单支持打开设置、立即提醒一次、退出应用。
   - 双击托盘图标打开设置窗口。

2. 喝水提醒配置
   - 默认提醒间隔为 30 分钟。
   - 用户可修改提醒间隔，单位为分钟。
   - 用户可开启或关闭喝水提醒。
   - 用户可开启或关闭开机自启动。
   - 用户可修改弹窗自动关闭时长，默认 15 秒。
   - 保存后立即生效，并持久化到 JSON 文件。

3. 提醒弹窗
   - 到达提醒时间时在屏幕右下角显示。
   - 弹窗显示随机提醒标题、随机正文、关闭按钮。
   - 文案根据当前时间段动态切换，可使用搞笑、逗趣、轻松的表达方式，但不允许出现悲伤语气。
   - 文案可参考近期鼠标点击活跃度，适当判断当前更像是深度专注还是高频处理状态。
   - 用户未操作时，倒计时结束自动关闭。
   - 用户点击关闭后立即消失，不影响下一轮提醒。

4. 状态展示
   - 设置窗口展示：
     - 当前提醒是否启用
     - 当前提醒间隔
     - 当前自动关闭秒数
     - 当前是否开机自启动
     - 上次提醒时间
     - 下次预计提醒时间
     - 最近保存状态
   - 当配置文件不存在时，以默认值初始化并提示“已加载默认配置”。

5. 现代化 UI 表现
   - 主界面左上角必须展示应用图标、应用名称和一句简短描述。
   - 首页使用至少一块核心状态卡片，强调“提醒中 / 已暂停 / 下次提醒时间”。
   - 重要操作按钮采用主次分明的视觉层级，不出现传统表单堆砌感。
   - 提醒弹窗加入柔和动效、玻璃感背景和大标题文案，整体观感偏现代而非传统 MessageBox。

#### 1.2.3 业务规则

- 提醒间隔允许范围：`5 - 240` 分钟。
- 自动关闭秒数允许范围：`3 - 15` 秒。
- 当提醒被关闭时，不再弹窗，但仍保留托盘图标和设置入口。
- 开机自启动默认为关闭，用户可独立控制，不受提醒总开关影响。
- 保存配置成功后，下一次提醒时间按“保存时刻 + 新间隔”重新计算。
- 立即提醒一次不会改写下一次正常提醒时间。
- 退出应用前必须释放 `NotifyIcon` 和所有定时器。
- 配置写入失败时，不更新内存中的已保存状态，并提示用户重试。

#### 1.2.4 关键状态

- `loading`：启动时读取配置，设置窗口进入加载态。
- `empty`：首次启动无配置文件，显示默认配置说明态。
- `error`：配置读取或保存失败，显示错误说明和重试按钮。
- `no-permission`：配置目录无法写入时，提示无权限并禁用保存按钮。

### 1.4 交互说明

#### 1.4.1 启动流程

1. 用户启动应用。
2. 应用检测单实例；若已有实例运行，则激活已有设置窗口或仅提示已在托盘运行。
3. 应用加载配置并创建托盘图标。
4. 若启动参数不要求打开界面，则主窗口不显示，直接进入后台运行。
5. 调度服务开始计时，等待下一次提醒。

#### 1.4.2 托盘交互流程

1. 用户右键托盘图标，打开菜单。
2. 选择“打开设置”后显示设置窗口。
3. 选择“立即提醒一次”后立即弹出提醒窗口。
4. 选择“暂停提醒 / 恢复提醒”后即时更新调度状态和托盘提示文本。
5. 选择“退出应用”后执行资源释放并结束进程。

#### 1.4.3 设置保存流程

1. 用户在设置窗口修改提醒开关、间隔或自动关闭时长。
2. 前端先执行表单校验：
   - 间隔必须为 5 至 240 的整数
   - 自动关闭必须为 3 至 15 的整数
   - 开机自启动为布尔开关，无额外数值校验
3. 校验通过后调用保存命令。
4. 保存成功：
   - 写入 JSON
   - 更新界面状态
   - 调度器重算下一次提醒时间
   - 弹出成功提示
5. 保存失败：
   - 保留用户输入
   - 显示错误提示
   - 不覆盖当前已生效配置

#### 1.4.4 提醒弹窗流程

1. 调度器检测到到期时间，触发提醒。
2. 弹窗从右下角进入，默认置顶但不抢占焦点。
3. 弹窗标题和正文根据当前时间段从内置文案池中随机生成。
4. 用户可以点击关闭按钮立即关闭。
5. 若无操作，则在倒计时结束后自动关闭。
6. 弹窗关闭后，调度器按提醒间隔计算下一次触发时间。
7. 若用户从托盘选择“立即提醒一次”，弹窗样式与常规提醒一致，但不重置正常调度时间。

#### 1.4.5 边界场景

- 首次运行无配置文件：自动生成默认配置并进入 empty 提示态。
- 配置文件损坏：回退到 error 态，允许用户重试或恢复默认值。
- 系统时间被修改：下一轮检查按“当前时间 >= 下次提醒时间”重新判断。
- 设置窗口被关闭：应用继续后台运行，不退出。
- 用户禁用提醒：清除待触发提醒，仅保留托盘入口。
- 用户临时暂停提醒：保留已保存配置，但暂停当前调度。
- 无权限写入配置目录：进入 no-permission 态，允许只读查看当前内存配置。

#### 1.4.6 错误处理方式

- 配置读取失败：提示“无法读取提醒配置”，提供重试。
- 配置保存失败：提示“保存失败，当前设置未生效”，保留编辑内容。
- 托盘初始化失败：记录日志并中止启动，避免进入不可退出状态。
- 弹窗创建失败：记录日志，保留下一轮提醒，不让调度器崩溃。
- 托盘库异常释放失败：写日志后继续退出流程，避免进程残留。

## 2. 基于 Avalonia 的重写方案

### 2.1 架构设计

#### 2.1.1 重写目标

- 在不改变当前业务能力的前提下，将项目从 `WPF + Wpf.Ui` 重写为 `Avalonia`。
- 保留现有分层思路：`View → ViewModel → Service → Repository → Model`。
- 优先复用已稳定的 DTO、持久化结构、提醒调度规则、随机文案逻辑和开机自启动业务规则，避免把“UI 迁移”扩大成“业务重做”。
- 首个交付目标仍以 `Windows` 桌面端可用为准；Avalonia 的跨平台能力只作为后续扩展余量，本阶段不强制同时交付 macOS / Linux。

#### 2.1.2 模块划分

- `src/MiniNotifier.App`
  - Avalonia 应用宿主。
  - 包含 `App.axaml`、`Program.cs`、窗口视图、样式资源、图标资源、托盘定义。
- `src/MiniNotifier.Core`
  - 复用并承接现有 `ViewModels`、`Services`、`Repositories`、`Models`、`Helpers`。
  - 不依赖 WPF/Avalonia 具体控件类型，保证核心逻辑可测试、可迁移。
- `src/MiniNotifier.Tests`
  - `xUnit` 业务测试。
  - `Avalonia.Headless.XUnit` 视图绑定与关键交互测试。

#### 2.1.3 Avalonia 版本与技术决策

- UI 框架采用 `Avalonia 12` 稳定版，主题基于 `Avalonia.Themes.Fluent`，不额外引入新的平行设计体系。
- MVVM 继续使用 `CommunityToolkit.Mvvm`，保持 `[ObservableProperty]` 与 `[RelayCommand]` 约定不变。
- 依赖注入继续使用 `Microsoft.Extensions.Hosting` 与 `Microsoft.Extensions.DependencyInjection`。
- 系统托盘改用 Avalonia 官方 `TrayIcon + NativeMenu`，替换当前 `Hardcodet.NotifyIcon.Wpf`。
- 提醒弹窗继续使用自定义窗口实现，保留右下角浮出、倒计时自动关闭、手动关闭等能力。
- 持久化继续使用 `System.Text.Json` 与现有 JSON 文件结构，配置文件路径保持 `%LocalAppData%/MiniNotifier/hydration-settings.json` 不变。
- Windows 开机自启动继续通过注册表服务实现，但抽象仍保留在 `IAutoStartService`，避免 UI 框架与平台逻辑耦合。
- 发布链路以 `Native AOT + Trim` 为目标，使用 SDK 支持的 `TrimmerRootDescriptor (Roots.xml)` 维护反射/序列化保留清单；后续如果引入 `App.config`、`HttpClient`、`Dapper` 或其他反射重依赖能力，统一在该文件补充保留规则。
- 为避免发布命令分散在人工步骤中，Avalonia 主工程需要提供固定的 `PublishProfile` 作为本地和 CI 复用入口；首个目标档位为 `win-x64 Native AOT`。
- 样式资源从当前 `ResourceDictionary` 迁移到 Avalonia `Styles/ResourceDictionary` 体系，颜色、间距、圆角仍由 `design-system.yaml` 中的 token 映射，不允许在控件属性中散落硬编码。

#### 2.1.4 数据流向

1. 应用通过 `Program.BuildAvaloniaApp()` 启动，并创建 `Generic Host`。
2. `App.axaml` 加载 Fluent 主题与自定义样式资源，同时注册应用级 `TrayIcon`。
3. `IHydrationSettingsService` 读取本地 JSON 配置；若不存在，则回填默认值并进入 `empty` 状态。
4. `IReminderSchedulerService` 根据当前配置启动调度。
5. 用户通过托盘点击或菜单打开主设置窗口；窗口默认只在需要时显示，不随应用启动自动弹出。
6. 用户保存设置后，`Service` 写入 JSON，成功后刷新调度器与托盘状态。
7. 调度器到期或用户点击“立即提醒”时，由提醒弹窗服务创建 Avalonia 浮窗。
8. 浮窗关闭后仅更新提醒时序，不中断应用后台运行。

#### 2.1.5 迁移策略

- 先搭 Avalonia 宿主、主题资源、托盘、窗口基础骨架。
- 再迁移设置窗口视图和现有 DTO 绑定。
- 再接入提醒弹窗、调度服务、配置持久化、自启动服务。
- 最后替换发布链路、补齐测试并移除 WPF 旧实现。
- 在 Avalonia 版本达到功能对齐前，旧 WPF 代码允许暂时保留在分支中作为对照，不在中途强行删除。
- 发布链路切换时，优先修改现有 CI 工作流去复用 Avalonia `PublishProfile`，避免本地命令、项目文件和 CI 参数三处漂移。

### 2.2 功能需求

#### 2.2.1 页面与入口

| 页面 / 入口 | 说明 |
| ----------- | ---- |
| 应用级托盘图标 | 应用主入口，支持点击打开设置、立即提醒、暂停/恢复提醒、退出应用 |
| 主设置窗口 | 展示当前提醒状态、基础配置、保存操作、测试提醒、错误与空状态 |
| 提醒弹窗 | 右下角浮出提醒，展示标题、正文、倒计时、关闭操作 |

#### 2.2.2 功能点明细

1. 托盘与应用生命周期
   - 应用启动后默认驻留托盘，不主动弹出主窗口。
   - 托盘悬停文案要反映当前提醒状态。
   - 左键打开设置窗口，右键打开原生菜单。
   - 菜单支持“打开设置”“立即提醒一次”“暂停/恢复提醒”“退出应用”。
2. 设置窗口
   - 左上角必须显示应用图标与应用名称。
   - 展示提醒启用状态、暂停状态、提醒间隔、自动关闭秒数、自启动状态、上次提醒、下次提醒、最近保存状态。
   - 支持立即提醒、保存设置、暂停/恢复提醒。
   - 支持 `loading / empty / error / no-permission` 完整页面状态。
3. 提醒配置
   - 用户可编辑提醒开关、提醒间隔、自启动、自动关闭秒数。
   - 保存后必须真实写入本地配置并立即生效，重新打开应用仍可回显。
4. 提醒弹窗
   - 按调度时间或手动测试触发。
   - 根据时间段和工作强度生成轻松、俏皮但不过度冒犯的文案。
   - 支持自动关闭和手动关闭。
5. 数据与兼容
   - 继续兼容现有 JSON 配置结构，避免因为 UI 框架替换导致用户配置丢失。
   - 继续兼容现有 Windows 开机自启动行为。

#### 2.2.3 业务规则

- 提醒间隔允许范围：`1 - 240` 分钟。
- 自动关闭秒数允许范围：`3 - 15` 秒。
- “立即提醒一次”不会重置正常调度时间。
- 关闭设置窗口不退出应用，仅隐藏窗口。
- 禁用提醒后停止后续自动弹窗，但保留托盘入口。
- 暂停提醒后保留当前配置，恢复后按最新配置重新计算下次提醒时间。
- 配置写入失败时，不覆盖内存中最后一次成功保存的状态。
- 托盘初始化失败属于阻断性错误，应用不得进入无法操作的半启动状态。

#### 2.2.4 关键状态

- `loading`：应用首次加载配置、托盘状态同步中。
- `empty`：本地尚无配置文件，已生成默认配置但需要用户确认或继续编辑。
- `error`：配置读取失败、弹窗初始化失败、托盘初始化失败后的可恢复状态。
- `no-permission`：无权限写入配置目录或更新开机自启动时进入只读提示态。

### 2.4 交互说明

#### 2.4.1 启动流程

1. 用户启动应用。
2. Avalonia 应用加载主题、资源和托盘定义。
3. `Generic Host` 初始化核心服务、仓储、ViewModel。
4. 配置加载成功后启动调度器；若失败则记录日志并进入可恢复状态。
5. 默认不主动显示主窗口，仅保留托盘入口。

#### 2.4.2 托盘交互流程

1. 用户左键托盘图标，打开或激活主设置窗口。
2. 用户右键托盘图标，显示原生菜单。
3. 点击“立即提醒一次”后立即创建提醒弹窗，但不改变正常调度时间。
4. 点击“暂停提醒 / 恢复提醒”后即时刷新托盘文案、窗口状态和调度器。
5. 点击“退出应用”后按顺序释放托盘、调度器、窗口与宿主资源。

#### 2.4.3 设置保存流程

1. 用户修改配置项。
2. ViewModel 先完成本地校验。
3. 校验通过后调用 `IHydrationSettingsService.SaveAsync()`。
4. 保存成功后：
   - 持久化 JSON。
   - 更新托盘文案。
   - 刷新下次提醒时间。
   - 显示成功反馈。
5. 保存失败后：
   - 保留用户当前输入。
   - 显示错误提示。
   - 不覆盖最后一次成功保存状态。

#### 2.4.4 提醒弹窗流程

1. 调度器到期后请求弹窗服务创建提醒窗口。
2. 弹窗从右下角滑入，默认不抢占主焦点。
3. 文案服务根据时间段和工作强度生成标题与正文。
4. 用户可以手动关闭；若无操作，则倒计时结束自动关闭。
5. 弹窗关闭后通知调度器计算下一次提醒时间。

#### 2.4.5 边界场景

- 首次运行无配置文件：自动创建默认配置并展示 `empty` 状态说明。
- 配置文件损坏：进入 `error` 状态，允许重试或恢复默认配置。
- 无权限写配置：进入 `no-permission` 状态，保留只读展示。
- 托盘图标不可用：视为启动失败，记录日志并阻止进入半残状态。
- 用户重复打开窗口：若窗口已存在，则激活并置前，而不是重复创建。
- 关闭主窗口：仅隐藏，不退出应用。

#### 2.4.6 错误处理方式

- 配置读取失败：提示“无法读取提醒配置”，提供重试入口。
- 配置保存失败：提示“保存失败，当前设置未生效”，保留编辑内容。
- 托盘初始化失败：写日志并终止启动。
- 弹窗创建失败：写日志，不影响下一轮提醒调度。
- 平台功能失败：如自启动更新失败，仅影响对应功能，不应拖垮主窗口与提醒流程。
