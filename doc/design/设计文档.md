## 1. 喝水提醒托盘应用

### 1.1 架构设计

#### 1.1.1 模块划分

- `App` 启动层：负责单实例启动、`Generic Host`、依赖注入、托盘组件初始化、窗口生命周期管理。
- `Views`
  - `HydrationSettingsWindow`：唯一主操作窗口，用于查看和修改提醒设置，采用现代化 Fluent 卡片布局。
  - `HydrationReminderWindow`：右下角提醒弹窗，支持手动关闭和自动关闭，采用轻玻璃感浮层样式。
- `ViewModels`
  - `HydrationSettingsViewModel`：维护设置页状态、保存设置、立即测试提醒。
  - `HydrationReminderViewModel`：维护提醒文案、倒计时显示、关闭动作。
- `Services`
  - `ITrayService`：托盘图标、菜单、双击打开设置、退出应用。
  - `IReminderSchedulerService`：基于 `DispatcherTimer` 进行提醒调度。
  - `IReminderPopupService`：统一创建和关闭提醒弹窗。
  - `ISettingsService`：读写提醒配置，负责默认值回填。
  - `IReminderMessageService`：根据时间段和内置文案池随机生成提醒标题与正文。
  - `IMouseActivityService`：统计近期全局鼠标点击活跃度，用于辅助推断当前工作状态。
- `Repositories`
  - `ISettingsRepository`：基于 JSON 文件持久化提醒设置。
- `Models`
- `HydrationSettings`：提醒间隔、弹窗自动关闭秒数、启用状态、最近提醒时间等领域模型。
- `HydrationSettingsDto`：阶段 1 输出的稳定契约。
- `HydrationSettingsJsonContext`：`System.Text.Json` source generation 上下文，保障发布兼容性。
  - `StartupSettingsDto`：开机自启动设置契约，供阶段 2 接入真实系统能力。

#### 1.1.2 数据流向

1. 应用启动后创建托盘图标，不主动显示主窗口。
2. `ISettingsService` 读取本地 JSON 配置；若文件不存在，则回填默认配置：
   - 喝水提醒开启
   - 提醒间隔 30 分钟
   - 弹窗自动关闭 15 秒
3. `IReminderSchedulerService` 使用 `DispatcherTimer` 按“下一次提醒时间”动态调度，而不是固定 1 秒轮询。
4. 到达提醒时间后，由 `IReminderPopupService` 打开右下角提醒弹窗。
5. 用户可通过弹窗关闭按钮关闭，也可等待 15 秒自动关闭。
6. 用户在设置窗口修改配置并保存后，`ISettingsService` 写入 JSON，调度服务即时重载。
7. 用户通过托盘菜单选择“退出应用”时，先释放计时器和托盘资源，再退出进程。

#### 1.1.3 技术决策

- 仅面向 Windows，采用 `.NET 10 + WPF + Wpf.Ui + CommunityToolkit.Mvvm`。
- 使用 `Microsoft.Extensions.Hosting` 的 `Generic Host` 管理 DI、配置和应用生命周期。
- 托盘采用 `Hardcodet.NotifyIcon.Wpf`，以获得更贴近 WPF 的托盘集成能力。
- 定时调度使用 `DispatcherTimer`，但按“距离下次提醒剩余时间”动态设置 Tick 间隔，减少空轮询。
- 提醒弹窗使用独立 WPF Window，自带显示动画和自动关闭计时器，优先保证样式可控和视觉一致性。
- 配置持久化采用 `System.Text.Json + source generation`，文件存放于 `%LocalAppData%/MiniNotifier/hydration-settings.json`。
- 开机自启动采用当前用户注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`，避免要求管理员权限。
- 提醒文案采用内置多套模板池，按晨间 / 午间 / 下午 / 傍晚 / 深夜等时间段随机生成，风格允许轻松搞笑，但不使用悲伤语气。
- 在不接入天气和节假日能力的前提下，可基于近期鼠标点击频率推断“深度专注 / 平稳推进 / 高频处理”等工作状态，用于调整提醒语气。
- 发布采用“框架依赖 + 单文件”模式，优先减少发布包体积；本阶段不启用 Trim，避免与 framework-dependent 模式冲突。
- 为避免重复提醒和资源浪费，应用需支持单实例运行。

#### 1.1.4 视觉与交互风格

- 整体视觉采用 `Fluent + Glass + Soft Glow` 方向，而不是传统系统设置页。
- 设置窗口使用顶部品牌区、中央主卡片、底部辅助说明的三段式布局，强调信息层次。
- 弹窗采用圆角浮层、轻微透明背景、柔和描边和右下角滑入动效，营造“轻提醒”体验。
- 重点信息用大号数字和状态标签呈现，例如“30 分钟”“15 秒自动关闭”“下一次 19:30”。
- 动效保持克制：
  - 窗口进入使用 `180-220ms` 位移 + 透明度过渡
  - 表单状态切换使用淡入淡出
  - 保存成功使用轻量 `Snackbar`
- 现代化不等于复杂化，所有主要操作必须在单窗口内完成，不新增无意义层级。

### 1.2 功能需求

#### 1.2.1 页面与入口

| 页面 / 入口 | 说明 |
| ----------- | ---- |
| 系统托盘图标 | 应用主入口，启动后仅显示托盘图标，不在任务栏保留主窗口 |
| 设置窗口 | 查看与修改喝水提醒配置，展示当前状态、下次提醒时间和现代化状态卡片 |
| 提醒弹窗 | 右下角浮出提醒窗口，展示喝水提示、倒计时和关闭操作 |

#### 1.2.2 功能点明细

1. 后台与托盘
   - 应用启动后默认最小化到托盘，不主动弹出主窗口。
   - 托盘图标悬停时显示应用名称与当前提醒状态。
   - 托盘菜单支持打开设置、立即提醒一次、退出应用。
   - 双击托盘图标打开设置窗口。

2. 喝水提醒配置
   - 默认提醒间隔为 30 分钟。
   - 用户可修改提醒间隔，单位为分钟。
   - 用户可开启或关闭喝水提醒。
   - 用户可开启或关闭开机自启动。
   - 用户可修改弹窗自动关闭时长，默认 15 秒。
   - 保存后立即生效，并持久化到 JSON 文件。

3. 提醒弹窗
   - 到达提醒时间时在屏幕右下角显示。
   - 弹窗显示随机提醒标题、随机正文、关闭按钮。
   - 文案根据当前时间段动态切换，可使用搞笑、逗趣、轻松的表达方式，但不允许出现悲伤语气。
   - 文案可参考近期鼠标点击活跃度，适当判断当前更像是深度专注还是高频处理状态。
   - 用户未操作时，倒计时结束自动关闭。
   - 用户点击关闭后立即消失，不影响下一轮提醒。

4. 状态展示
   - 设置窗口展示：
     - 当前提醒是否启用
     - 当前提醒间隔
     - 当前自动关闭秒数
     - 当前是否开机自启动
     - 上次提醒时间
     - 下次预计提醒时间
     - 最近保存状态
   - 当配置文件不存在时，以默认值初始化并提示“已加载默认配置”。

5. 现代化 UI 表现
   - 主界面左上角必须展示应用图标、应用名称和一句简短描述。
   - 首页使用至少一块核心状态卡片，强调“提醒中 / 已暂停 / 下次提醒时间”。
   - 重要操作按钮采用主次分明的视觉层级，不出现传统表单堆砌感。
   - 提醒弹窗加入柔和动效、玻璃感背景和大标题文案，整体观感偏现代而非传统 MessageBox。

#### 1.2.3 业务规则

- 提醒间隔允许范围：`5 - 240` 分钟。
- 自动关闭秒数允许范围：`3 - 15` 秒。
- 当提醒被关闭时，不再弹窗，但仍保留托盘图标和设置入口。
- 开机自启动默认为关闭，用户可独立控制，不受提醒总开关影响。
- 保存配置成功后，下一次提醒时间按“保存时刻 + 新间隔”重新计算。
- 立即提醒一次不会改写下一次正常提醒时间。
- 退出应用前必须释放 `NotifyIcon` 和所有定时器。
- 配置写入失败时，不更新内存中的已保存状态，并提示用户重试。

#### 1.2.4 关键状态

- `loading`：启动时读取配置，设置窗口进入加载态。
- `empty`：首次启动无配置文件，显示默认配置说明态。
- `error`：配置读取或保存失败，显示错误说明和重试按钮。
- `no-permission`：配置目录无法写入时，提示无权限并禁用保存按钮。

### 1.4 交互说明

#### 1.4.1 启动流程

1. 用户启动应用。
2. 应用检测单实例；若已有实例运行，则激活已有设置窗口或仅提示已在托盘运行。
3. 应用加载配置并创建托盘图标。
4. 若启动参数不要求打开界面，则主窗口不显示，直接进入后台运行。
5. 调度服务开始计时，等待下一次提醒。

#### 1.4.2 托盘交互流程

1. 用户右键托盘图标，打开菜单。
2. 选择“打开设置”后显示设置窗口。
3. 选择“立即提醒一次”后立即弹出提醒窗口。
4. 选择“暂停提醒 / 恢复提醒”后即时更新调度状态和托盘提示文本。
5. 选择“退出应用”后执行资源释放并结束进程。

#### 1.4.3 设置保存流程

1. 用户在设置窗口修改提醒开关、间隔或自动关闭时长。
2. 前端先执行表单校验：
   - 间隔必须为 5 至 240 的整数
   - 自动关闭必须为 3 至 15 的整数
   - 开机自启动为布尔开关，无额外数值校验
3. 校验通过后调用保存命令。
4. 保存成功：
   - 写入 JSON
   - 更新界面状态
   - 调度器重算下一次提醒时间
   - 弹出成功提示
5. 保存失败：
   - 保留用户输入
   - 显示错误提示
   - 不覆盖当前已生效配置

#### 1.4.4 提醒弹窗流程

1. 调度器检测到到期时间，触发提醒。
2. 弹窗从右下角进入，默认置顶但不抢占焦点。
3. 弹窗标题和正文根据当前时间段从内置文案池中随机生成。
4. 用户可以点击关闭按钮立即关闭。
5. 若无操作，则在倒计时结束后自动关闭。
6. 弹窗关闭后，调度器按提醒间隔计算下一次触发时间。
7. 若用户从托盘选择“立即提醒一次”，弹窗样式与常规提醒一致，但不重置正常调度时间。

#### 1.4.5 边界场景

- 首次运行无配置文件：自动生成默认配置并进入 empty 提示态。
- 配置文件损坏：回退到 error 态，允许用户重试或恢复默认值。
- 系统时间被修改：下一轮检查按“当前时间 >= 下次提醒时间”重新判断。
- 设置窗口被关闭：应用继续后台运行，不退出。
- 用户禁用提醒：清除待触发提醒，仅保留托盘入口。
- 用户临时暂停提醒：保留已保存配置，但暂停当前调度。
- 无权限写入配置目录：进入 no-permission 态，允许只读查看当前内存配置。

#### 1.4.6 错误处理方式

- 配置读取失败：提示“无法读取提醒配置”，提供重试。
- 配置保存失败：提示“保存失败，当前设置未生效”，保留编辑内容。
- 托盘初始化失败：记录日志并中止启动，避免进入不可退出状态。
- 弹窗创建失败：记录日志，保留下一轮提醒，不让调度器崩溃。
- 托盘库异常释放失败：写日志后继续退出流程，避免进程残留。

## 2. 自动与手动检测更新

> 本章节参考 `C:\Work\github\ApixPress` 的更新检测链路：应用启动时按用户偏好静默检查新版；关于/设置区域提供手动检查入口；更新清单使用 `versions.json`；发现新版本后再由用户确认是否启动更新流程。

### 2.1 架构设计

#### 2.1.1 模块划分

- `Models/DTOs`
  - `AppUpdateCheckResultDto`：更新检测结果，包含当前版本、最新版本、是否有更新、发布时间、下载地址、包名与包校验值。
  - `AppUpdateManifestItemDto`：远端 `versions.json` 清单项契约，兼容 ApixPress 的 `PacketName`、`Hash`、`Version`、`Url`、`PubTime` 字段。
  - `AppUpdateLaunchRequestDto`：启动隐藏更新模式时写入临时目录的请求文件，记录下载地址、校验值、目标目录和重启程序路径。
- `Services`
  - `IApplicationUpdateService` / `ApplicationUpdateService`：负责读取配置中的更新源、请求远端清单、比较版本、返回结果，并在用户确认后启动更新流程。
  - `AppUpdateRunner`：复用主程序隐藏模式执行下载、SHA256 校验、解压、生成替换脚本、等待主进程退出并重启应用。
  - `IUpdateNotificationService`（可选轻量服务）：负责把自动检查结果转换为托盘气泡、设置页状态或 Wpf.Ui Snackbar。
- `ViewModels`
  - `MainWindowViewModel`：新增更新状态字段、自动检查开关、最近检查时间、最新版本、手动检查命令。
  - 启动后在 `InitializeAsync` 完成基础设置加载后触发一次静默自动检查；失败只记录日志，不打断提醒应用启动。
- `Views`
  - `MainWindow` 设置区域新增“软件更新”小节，包含自动检查开关、手动检查按钮、当前版本、最新版本、最近检查时间、更新状态。
  - 托盘菜单新增“检查更新”入口，便于不打开完整设置时手动触发。
- `Configuration`
  - `appsettings.json` 或内嵌默认配置新增 `Update` 节点：`ChannelName`、`VersionManifestUrl`。
  - 若仓库当前没有 appsettings 配置体系，则优先使用 `Microsoft.Extensions.Configuration` 既有 Host 能力读取内嵌默认值和外部覆盖文件，不新增平行配置体系。

#### 2.1.2 数据流向

1. 应用启动，Host 注册更新服务与 HTTP 客户端。
2. `MainWindowViewModel.InitializeAsync` 读取本地设置，若“自动检查更新”开启且更新源已配置，则后台调用 `IApplicationUpdateService.CheckForUpdatesAsync(currentVersion)`。
3. `ApplicationUpdateService` 从 `Update:VersionManifestUrl` 下载 `versions.json`，兼容数组格式与单对象格式。
4. 服务按发布时间与版本号选出最新有效版本，将 `1.0`、`1.0.0`、`1.0.0.0` 归一化比较，避免等价版本误报。
5. 若发现新版本，自动检查只提示“不自动安装”；用户点击设置页或托盘中的“立即更新”后，服务写入临时 `request.json` 并以隐藏更新模式启动当前主程序。
6. `AppUpdateRunner` 下载 zip 包、校验 SHA256、解压到临时目录，生成隐藏 `cmd` 脚本等待当前进程退出后覆盖目标目录并重启 `MiniNotifier.exe`。
7. 手动检查与自动检查复用同一服务；手动检查会在设置页展示完整状态，自动检查仅在发现新版本时显示轻提示。

#### 2.1.3 技术决策

- 版本清单格式与 ApixPress 保持一致，降低发布脚本和 GitHub Releases 产物的差异。
- 更新检测只做网络请求与版本比较，自动检查默认不下载、不替换，避免用户无感知修改本地程序。
- 更新包校验使用 SHA256；缺少 `Hash`、`Url` 或当前主程序路径不可识别时禁止启动更新。
- 隐藏更新模式复用主程序，不新增独立 updater exe，发布产物继续保持单主程序思路。
- 自动检查失败不阻塞喝水提醒、托盘初始化和设置窗口打开；错误记录到本地诊断日志，并在手动检查时展示可读错误。

### 2.2 功能需求

#### 2.2.1 页面与入口

| 页面 / 入口 | 说明 |
| ----------- | ---- |
| 设置窗口 - 软件更新 | 查看当前版本、最新版本、最近检查时间、更新源状态；支持手动检查和启动更新 |
| 设置窗口 - 自动检查开关 | 控制应用启动后是否自动检查更新，状态持久化到本地设置 |
| 托盘菜单 - 检查更新 | 不打开完整设置窗口时触发手动检查；检查结果通过托盘提示或设置窗口状态反馈 |
| 启动自动检查 | 应用启动后静默执行一次，发现新版时轻提示用户前往设置窗口处理 |

#### 2.2.2 功能点明细

1. 自动检测更新
   - 默认开启“启动时自动检查更新”。
   - 每次应用启动后最多执行一次自动检查。
   - 自动检查只在发现新版本时提示；无更新或检查失败不弹出干扰提示。
   - 若更新源未配置，则不执行自动检查。

2. 手动检测更新
   - 设置窗口提供“检查更新”按钮，执行期间显示“检查中...”并禁用重复点击。
   - 托盘菜单提供“检查更新”入口，触发同一业务逻辑。
   - 检查成功且无更新时提示“当前已是最新版本”。
   - 检查成功且有更新时展示最新版本、发布时间，并提供“立即更新”按钮。

3. 启动更新
   - 用户确认后才启动更新程序。
   - 启动前校验下载地址、包校验值、当前主程序路径和目标目录。
   - 更新程序启动成功后提示应用将关闭并更新；失败时保留当前应用运行。

4. 更新配置
   - 支持通过外部 `appsettings.json` 覆盖更新源。
   - 配置项至少包含：
     - `Update:ChannelName`
     - `Update:VersionManifestUrl`
   - 若未配置更新源，设置页展示“尚未配置更新源”，手动检查按钮仍可点击但返回明确错误。

#### 2.2.3 业务规则

- 版本比较使用 `System.Version` 归一化到四段比较；无法解析的清单版本跳过。
- 清单中存在多个版本时，优先按 `PubTime` 降序，再按版本号降序选择最新有效版本。
- 支持以下清单形态：
  - 单对象：`{ "PacketName": "...", "Version": "...", ... }`
  - 数组：`[{ "PacketName": "...", "Version": "...", ... }]`
- 当前版本号来自程序集 `InformationalVersion` 或 `Version`，去除 `+metadata` 后参与比较。
- 自动检查不主动关闭应用、不启动下载、不覆盖文件。
- 手动启动更新后，若主程序路径不存在、更新包地址为空、Hash 为空或启动隐藏更新模式失败，必须展示错误并停止。

#### 2.2.4 关键状态

- `loading`：手动检查中，按钮禁用，状态文案显示当前更新渠道。
- `empty`：尚未检查过，展示当前版本与“尚未检查”。
- `error`：网络失败、清单为空、清单解析失败、更新源未配置或启动更新失败。
- `no-permission`：更新脚本无法写入目标目录或临时目录时，提示用户检查安装目录权限。

### 2.4 交互说明

#### 2.4.1 自动检查流程

1. 用户启动应用。
2. 主设置加载完成后读取“启动时自动检查更新”开关。
3. 若开关关闭或更新源未配置，则跳过。
4. 若开关开启，则后台请求更新清单。
5. 无更新：只更新内部最近检查时间，不打扰用户。
6. 有更新：显示托盘/ Snackbar 提示“发现新版本”，点击提示或托盘“检查更新”可打开设置窗口查看详情。
7. 自动检查异常：写入本地诊断日志，不阻断启动。

#### 2.4.2 手动检查流程

1. 用户在设置页或托盘点击“检查更新”。
2. 界面进入 loading，按钮文字变为“检查中...”。
3. 服务返回失败时，展示失败原因和错误码。
4. 服务返回无更新时，展示当前已是最新版本并刷新最近检查时间。
5. 服务返回有更新时，展示最新版本、发布时间和“立即更新”按钮。

#### 2.4.3 启动更新流程

1. 用户点击“立即更新”。
2. 应用校验更新信息和当前运行路径。
3. 写入临时 `request.json`。
4. 使用当前 `MiniNotifier.exe --apply-update --request-file "request.json"` 启动隐藏更新模式。
5. 启动成功后关闭当前主窗口并退出应用；隐藏更新进程下载、校验、解压并生成替换脚本。
6. 替换脚本等待主程序和更新进程退出，覆盖目标目录后重启应用。

#### 2.4.4 边界场景

- 离线或 GitHub 不可访问：手动检查显示网络失败，自动检查静默记录。
- 清单为空：提示“更新清单为空”。
- 清单版本格式无效：跳过无效项；全部无效时提示“未找到有效版本”。
- 当前版本与最新版本等价：不提示更新。
- 发布包 Hash 缺失：禁止启动更新。
- 安装目录无权限：进入 no-permission 状态，提示检查安装目录权限。
